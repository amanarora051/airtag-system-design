Layer 1 — Edge Device (AirTag)

🏷️ IoT Beacon / Producer

Stateless, ultra-low-power edge node. Think of it as a lightweight microservice that only broadcasts events.

  • 📡 BLE Beacon Publisher — Broadcasts rotating EC P-224 public key every 2s (ADV_IND packets, 31 bytes)
  • 🎯 UWB Ranging Service — Time-of-flight distance measurement via Apple U1/U2 chip (~60ms pulse interval)
  • 📱 NFC Tag (Read-Only) — MIFARE Plus Type 4 tag serving a URL for Lost Mode identification
  • 🔑 Key Rotation Scheduler — Derives new key pair via KDF every ~15 min; full address rotation daily at 04:00
  • 📐 Accelerometer Polling — Samples every 10s (idle) or 0.5s (motion detected) for Lost Mode triggers
  • 🔊 Audio Alert DAC — Drives speaker coil (~8mA); plays sound on remote command or lost-mode motion
Layer 2 — Mesh Network (Find My)

🌐 Distributed Sensor / Relay Network

~1 billion Apple devices act as anonymous relay nodes. Analogous to a massive event-driven pub/sub mesh.

  • 📶 BLE Scanner (Background) — Every iPhone/iPad/Mac passively scans for FindMy beacon payloads
  • 📍 Location Sampler — Captures finder's own GPS/Wi-Fi position upon beacon detection
  • 🔐 Encrypt-and-Relay Agent — Encrypts location with AirTag's broadcast public key Pᵢ (ECIES), then relays encrypted blob upstream
  • 👤 Zero-Knowledge Identity — No auth headers, no user IDs, no logs. Finder is 100% anonymous to Apple
  • ⚡ Efficiency Guarantee — Uses <1KB per detection; piggybacks on existing network connections
Layer 3 — Cloud Backend (Apple iCloud)

☁️ Blind Relay / Encrypted Blob Store

Apple's servers are a "dumb pipe" — they store and forward encrypted location blobs but cannot decrypt them. Zero-knowledge architecture.

  • 📦 Encrypted Location Store — Key-value store keyed by hashed public key identifiers; values are E2E-encrypted location blobs
  • 🔀 Anonymous Proxy/Relay — All queries go through proxy servers that strip requester identity
  • 📨 Push Notification Service (APNs) — Triggers owner's device when new location reports arrive or Lost Mode AirTag is detected
  • 🔗 NFC URL Resolver — found.apple.com endpoint that resolves AirTag serial/public identity → owner's Lost Mode message
  • 🛡️ Anti-Stalking Detection — Distributed threat detection: alerts iPhone users if an unknown AirTag is moving with them
Layer 4 — Client App (Find My)

📱 Owner's Decryption & UI Client

The only entity that holds the private key. All decryption and rendering happens on-device.

  • 🗝️ Private Key Store — Root key pair synced via iCloud Keychain (E2E encrypted); derives all past/future keys locally
  • 📊 Location Decryptor — Fetches encrypted blobs from iCloud, decrypts with derived private key dᵢ → GPS coordinates
  • 🗺️ Map Renderer — Plots AirTag location history on Apple Maps with timestamps
  • 🧭 Precision Finding UI — On-device ML fuses UWB distance + IMU (accel/gyro) data → real-time directional arrow + haptics
  • 🔔 Alert Engine — "Left Behind" geofence alerts, "Unknown AirTag" stalking alerts, battery-low notifications
Cross-Cutting — Security & Privacy

🔒 Privacy-by-Design Guarantees

Privacy isn't an add-on; it's a hard architectural constraint across every layer.

  • 🔄 Rotating Identity — Public key rotates every ~15 min; BLE address changes daily → prevents persistent tracking
  • 🔏 End-to-End Encryption — ECIES with P-224; Apple is a blind relay. Even with full server access, locations are unreadable
  • 🚫 Zero-Knowledge Relaying — Finders send no auth info; Apple logs nothing linking finder ↔ owner
  • 🔊 Anti-Stalking Safeguards — Sound alerts after separation, iPhone alerts for unknown tags, Tracker Detect for Android
  • 🧮 Unlinkable Key Derivation — dᵢ = uᵢ·d + vᵢ prevents correlating Pᵢ ↔ Pⱼ without the root secret
1

🏷️ AirTag broadcasts BLE beacon

Emits 31-byte ADV_IND packet every 2 seconds containing rotating P-224 public key Pᵢ

Software analogy: A Kafka producer publishing events to a broadcast topic. The "topic ID" (public key) changes every 15 min so historical events can't be linked.

Packet structure: Bytes 0–5: Apple company ID + payload type | Bytes 6: Status | Bytes 7–29: 23 bytes of EC P-224 public key | Byte 30: upper 2 bits of first key byte | Byte 31: crypto counter (rotates every 15 min)

Key detail: The 6-byte BLE random static address is cleverly derived from the first 6 bytes of the public key itself, saving payload space.
2

📶 Nearby Apple device detects signal

Any of ~1 billion Find My-enabled devices passively scans for FindMy beacon payloads

Software analogy: A distributed fleet of consumer/subscriber nodes. Each node runs a background BLE scan service — like a sidecar agent in a service mesh.

Zero-knowledge: The finder device has no idea who owns the AirTag, what it's attached to, or whether it's in Lost Mode. It only sees an opaque public key.
3

🔐 Finder encrypts its GPS location with Pᵢ

ECIES encryption — only the owner's private key dᵢ can decrypt

Software analogy: Envelope encryption, like AWS KMS. The finder generates a one-time symmetric key, encrypts the location payload, then encrypts that symmetric key with the AirTag's public key Pᵢ. The result is an opaque ciphertext blob.

Privacy guarantee: Even the finder can't re-read its own report after sending it — the ephemeral key is discarded.
4

☁️ Encrypted blob sent to Apple's iCloud

Blob is stored in a key-value store keyed by a hash of Pᵢ — Apple sees nothing

Software analogy: A blind message broker / encrypted object store (like S3 with client-side encryption). Apple is the "dumb pipe" — it can route and store blobs but never decrypt them.

No auth headers: The upload request contains zero authentication. Apple doesn't know who the finder is, and retains no logs correlating finder ↔ owner.
5

📱 Owner queries iCloud for their AirTag

Find My app derives the current Pᵢ locally, queries iCloud via anonymous proxy

Software analogy: The client re-derives the "partition key" from its local key material and issues a lookup — like querying a DHT (distributed hash table) where you need the key to find AND read the value.

Anonymous proxy: All queries pass through relay servers that strip the requester's identity. Apple cannot correlate who is asking for which key.
6

🗝️ Owner decrypts location on-device

Private key dᵢ (derived from root secret via KDF chain) decrypts → GPS coordinates

Software analogy: Client-side decryption like Signal Protocol's "ratcheting" — each time period has a unique derived key, ensuring forward secrecy. Only the owner's device, which holds the root secret synced via iCloud Keychain, can derive any dᵢ.

Precision Finding (within ~30ft): If the owner is close enough, UWB ranging activates. On-device ML fuses UWB time-of-flight data with IMU sensors (accelerometer + gyroscope) to render a real-time directional arrow and distance, with haptic feedback intensifying on approach.

1. Initialization (Registration)

When AirTag is paired, the owner's iPhone generates:

EC key pair: {d, P} where P = d·G on curve P-224 Secret: SK₀ ← random(256 bits) Counter: i = 0

The key pair and SK₀ are synced across the owner's devices via iCloud Keychain (E2E encrypted). They are never sent to Apple.

2. Key Rotation (~every 15 minutes)

Derive the current symmetric key and then a short-lived key pair:

SKᵢ = KDF(SKᵢ₋₁, "update") // ratchet forward (uᵢ, vᵢ) = KDF(SKᵢ, "diversify") // derive affine coefficients dᵢ = uᵢ · d + vᵢ (mod n) // derived private key Pᵢ = uᵢ · P + vᵢ · G // derived public key (broadcast in BLE)

Unlinkability: Given Pᵢ and Pⱼ (i ≠ j), it is computationally infeasible to determine they belong to the same device without knowing d or SK₀.

3. Location Encryption (by Finder)

The detecting device performs ECIES (Elliptic Curve Integrated Encryption Scheme):

1. Generate ephemeral key pair: {e, E = e·G} 2. Shared secret: S = e · Pᵢ // ECDH with AirTag's public key 3. Derive symmetric key: K = KDF(S) 4. Encrypt: C = AES-GCM(K, location || timestamp) 5. Send to Apple: { hash(Pᵢ), E, C } // no auth, no identity

The finder discards the ephemeral key e immediately — it can never re-read its own report.

4. Location Decryption (by Owner)

The owner's device re-derives the matching private key and decrypts:

1. Re-derive dᵢ from root secret: SKᵢ → (uᵢ, vᵢ) → dᵢ 2. Shared secret: S = dᵢ · E // ECDH with finder's ephemeral public key 3. Derive symmetric key: K = KDF(S) 4. Decrypt: location || timestamp = AES-GCM⁻¹(K, C) 5. Plot on map in Find My app

Forward secrecy: Compromising a future SKᵢ doesn't reveal past keys due to the one-way KDF ratchet.

AirTag Operating States (Firmware State Machine)

Not Registered
Brand new or factory-reset. Advertises at 33ms interval awaiting pairing. Uses default BT address.
↓ owner pairs via iPhone (one-tap setup)
Initialization
EC P-224 key pair generated. Root secret SK₀ shared between AirTag and iPhone. Registration to Apple ID.
↓ key exchange complete
Connected
Owner's device in BLE range. No broadcasts occur — silent mode. BLE connection interval: 1s. Precision Finding available via UWB.
↓ owner leaves BLE range
Disconnected
Broadcasts rotating public key Pᵢ every 2 seconds. New identity generated upon disconnection. Find My network can now locate it.
↓ after ~3 days separated
Lost
Enters "Waiting for Motion" — samples accelerometer every 10s. On motion → plays 20s sound alert, then silent for 6 hours. Cycle repeats.
↓ motion detected while in Lost
Sound Alert
Speaker plays alert sound for ≤20s of detected motion (~8mA draw). Can also be triggered remotely by owner via Find My. Overrides Precision Finding.
↓ owner comes back in range
Precision Finding
UWB ranging activated by owner's U1/U2-equipped iPhone. Transmits UWB pulses every ~60ms. iPhone ML fuses ToF + IMU → directional guidance with haptics.

AirTag ↔ Software System Mapping

Every AirTag component maps cleanly to a familiar distributed systems concept.

AirTag Component Software Analogy Why
AirTag device Stateless IoT event producer Only emits events (BLE beacons); holds no location data itself. Like a Lambda function that fires and forgets.
BLE beacon broadcast Pub/Sub topic publish Broadcasts to an open "topic" that any subscriber in range can pick up — like Kafka produce with no ACK.
~1B Apple devices Distributed mesh / CDN edge nodes Globally distributed passive relay fleet. Each node detects, enriches (adds GPS), encrypts, and forwards — like CDN edge servers or a service mesh sidecar fleet.
Rotating P-224 public key Rotating API key / JWT with short TTL Identity rotates on a schedule to prevent correlation. Like short-lived tokens in OAuth with refresh rotation.
ECIES location encryption Envelope encryption (AWS KMS style) Encrypt data with a symmetric key, then encrypt that key with the recipient's public key. Finder generates ephemeral key → ECDH → AES-GCM.
Apple iCloud backend Blind relay / encrypted object store Stores opaque blobs keyed by hash(Pᵢ). Cannot decrypt contents. Like S3 with client-side encryption + zero-knowledge access.
iCloud Keychain sync Distributed secret manager Syncs root key material across owner's devices with E2E encryption — like HashiCorp Vault replication with transit encryption.
KDF key ratchet Signal Protocol double ratchet One-way KDF chain ensures forward secrecy. Compromising future key doesn't reveal past keys.
Precision Finding (UWB + ML) Edge ML inference pipeline Sensor fusion (UWB ToF + accelerometer + gyroscope) processed by on-device ML model → real-time UI. All inference local, zero network latency.
Anti-stalking alerts Distributed anomaly detection Each iPhone runs a local anomaly detector: "is an unknown AirTag correlating with my movement over time?" Like a fleet-wide IDS/IPS system.
Lost Mode NFC tap QR code → deep link resolver NFC tap serves a URL (found.apple.com) that resolves to owner's contact info. Like scanning a QR code that opens a specific API endpoint.
Find My app Thick client / offline-first app All crypto and rendering happens client-side. Server is just a blob store. Like an offline-first app with local-first data and sync.
🏗️ Architecture pattern: AirTag is fundamentally a zero-knowledge, crowdsourced, event-driven location system — combining patterns from pub/sub messaging, client-side encryption, distributed hash tables, sensor fusion ML at the edge, and anonymous relay networks (like Tor's onion routing, but for location data).
